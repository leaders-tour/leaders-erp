# 📘 PRD v2

## Tour Itinerary Admin – TimeTable 기반 여행 일정 ERP

---

# 1. 제품 개요

## 1.1 배경

기존 여행 일정은 디자인 툴(예: 미리캔버스)에서 수작업으로 제작되고 있다.  
이 방식은 다음과 같은 문제를 가진다:

- 반복 작업 과다
- 일정 간 일관성 부족
- 거리·시간 계산 수작업 의존
- 일정 수정 시 재디자인 비용 발생
- 상품 버전 관리 어려움

이를 해결하기 위해:

> 목적지 선택만으로 자동 일정이 생성되는  
> “Graph 기반 여행 일정 생성 ERP”를 구축한다.

---

# 2. 제품 목표

## 2.1 1차 목표 (MVP)

- 순차적 목적지 선택 기반 일정 생성
- 시간-일정 통합 타임테이블 구조
- 이동 거리 자동 계산 구조
- 숙소 및 식사 자동 매핑
- 일부 항목 수동 수정 가능
- 인쇄/PDF 출력 가능

※ JSON 관련 기능은 내부 개발 디버깅 요소로 제외

# 5. 시간-일정 설계 (핵심 설계 철학)

## 5.1 왜 구조를 변경했는가

기존 일정 구성 방식은:

- 시간 컬럼과 일정 컬럼이 분리되어 있었고
- 두 정보가 서로 연관되어 있음에도 독립적으로 작성되었다.

문제점:

- 시간과 일정이 매칭되지 않는 오류 발생 가능
- 일정이 수정될 경우 시간 동기화가 어렵다
- PDF 변환 시 시각적 정렬이 불명확
- 데이터 구조상 확장성이 낮다

---

## 5.2 시간-일정 통합 구조

현재는 시간과 일정이 분리된 텍스트가 아니라

> “시간 기반 이벤트 리스트”

형태로 설계되어 있다.

각 일정 항목은:

- 시간
- 해당 시간에 수행되는 활동

을 하나의 단위로 묶는다.

이 구조는 다음을 보장한다:

- 시간과 활동의 1:1 연결
- 순서 기반 일정 흐름 보장
- 일정 재정렬 가능
- Drag & Drop 확장 가능
- PDF 타임라인 변환 용이

---

## 5.3 타임테이블의 의미

타임테이블은 단순한 표가 아니라:

> “하루의 흐름을 정의하는 구조화된 이벤트 시퀀스”

이다.

이로 인해:

- 이동 시간과 활동이 자연스럽게 연결됨
- 식사/휴식/투어 구간이 명확히 구분됨
- 사용자 경험 흐름 기반 설계 가능

---

# 6. 루트 설계 방식

## 6.1 순차 선택 구조

일정은 한 번에 모든 일차를 선택하는 방식이 아니다.

1일차 선택  
→ 해당 목적지에서 갈 수 있는 2일차 후보만 노출  
→ 선택  
→ 다음 후보 노출

이 구조는:

- 불가능한 경로 선택 방지
- 상품 구조 표준화
- 운영자 실수 방지
- 그래프 기반 확장 가능성 확보

---

# 7. 자동화 영역

## 7.1 이동 정보

- 목적지 간 평균 거리
- 예상 이동 시간

자동 표기되며  
향후 실시간 거리 API로 교체 가능

---

## 7.2 숙소 자동 매핑

각 목적지에는 기본 숙소 타입이 연결되어 있다.

예:

- 여행자 캠프
- 온천 리조트
- 숙소 미포함

---

## 7.3 식사 규칙

다음 조건에 따라 자동 설정된다:

- 숙소 유형
- 출발 타입
- 마지막 일정 여부
